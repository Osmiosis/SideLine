# Backend Engineering Spec — Operator App (for Claude Code)
**Product:** A LOCAL web app that lets a non-technical operator turn a recorded match into deliverables with zero code/terminal. This spec = the BACKEND: serves the UI, collects human inputs, runs the EXISTING CV pipelines as background jobs, returns deliverables.
**Architecture (confirmed):** LOCAL laptop server (the RTX 4060 machine) on the LAN. Teacher's browser (phone/laptop on same wifi) hits the laptop's local server. NO cloud. No recurring cost, footage never leaves the school (privacy intact).
**Stack (confirmed):** FastAPI backend (Python — same language as the entire pipeline, so it CALLS the existing scripts directly). Frontend built separately in a design tool; backend serves it as static files + a JSON API.

---

## THE CORE PRINCIPLE (read first — this governs everything)
The backend is a THIN ORCHESTRATION LAYER. It does NOT reimplement ANY CV/ML logic. It:
1. Collects human inputs from the UI,
2. Writes them to a JOB CONFIG FILE,
3. Invokes the EXISTING pipeline scripts (detection, tracking, analytics, event/highlight clipping) as background jobs that READ that config,
4. Collects the output files and serves them back.

This is exactly the app-readiness habit already followed: every human step produces a SAVED FILE the pipeline consumes. The backend formalizes that. **Do NOT touch or duplicate the CV code — wrap it.**

---

## The job-config-file contract (the heart of the system)
Every match = a JOB with a directory and a `job_config.json`. The UI collects inputs → backend writes this → pipelines read it. This is the SAME contract the frontend design brief collects against (keep field names aligned).

```
jobs/<job_id>/
  job_config.json
  raw_video.mp4              # uploaded footage
  calibration.json           # court/pitch marked points
  roster.json                # player names
  tags.json                  # clip_id -> player name (player highlights only)
  status.json                # job state machine (see below)
  outputs/                   # finished deliverables land here
```

`job_config.json` shape:
```json
{
  "job_id": "uuid",
  "sport": "football | basketball",
  "match_name": "string",
  "match_date": "YYYY-MM-DD",
  "video_path": "raw_video.mp4",
  "calibration_points": [{"pixel_x": int, "pixel_y": int, "real_world_label": "string"}],
  "roster": ["name", ...],
  "player_tags": {"clip_id": "player_name"},
  "deliverables_requested": ["coach_analytics", "event_highlights", "player_highlights"],
  "created_at": "iso8601"
}
```
The existing pipeline scripts get adapted (minimally) to accept this config as their input source instead of hardcoded paths/prompts. (This is the SoccerNet-decoupling work started on Day-31, finished here properly.)

## API endpoints (FastAPI, JSON)
- `POST /api/jobs` — create a job (sport, name, date) → returns job_id
- `POST /api/jobs/{id}/video` — upload footage (chunked/streamed; files are large — support resumable or at least streamed upload, show progress)
- `GET  /api/jobs/{id}/frame` — return a still frame from the video (for the court-marking screen)
- `POST /api/jobs/{id}/calibration` — save calibration_points
- `POST /api/jobs/{id}/roster` — save roster
- `GET  /api/jobs/{id}/tagging-clips` — list clips needing tags (player highlights; generated by an early pipeline pass)
- `POST /api/jobs/{id}/tags` — save player_tags
- `POST /api/jobs/{id}/deliverables` — set deliverables_requested + START processing (enqueue job)
- `GET  /api/jobs/{id}/status` — current state + stage + progress (UI polls this)
- `GET  /api/jobs/{id}/outputs` — list finished deliverable files
- `GET  /api/jobs/{id}/outputs/{file}` — download a deliverable
- `GET  /api/jobs` — list all jobs (dashboard)
- Serve the static frontend at `/`

## The job queue / async model (critical — matches take HOURS)
- A web request must NEVER block on processing. Submit → return immediately → process in background → UI polls `/status`.
- **Keep it simple — single-laptop, one-match-at-a-time, single PE department.** Do NOT build heavyweight infra (no Celery/Redis cluster). A lightweight in-process background worker + a job queue table is enough:
  - A `status.json` (or small SQLite) per job: `queued | processing | ready | failed`, current `stage`, `progress`, `error`.
  - A single background worker loop that picks the next `queued` job and runs the pipeline stages sequentially (GPU = one job at a time anyway).
  - FastAPI `BackgroundTasks` or a simple `asyncio`/threading worker; SQLite for job state if you want durability across restarts.
- **Stage-based progress** (the UI shows these as plain English): e.g. `uploading → calibration_pending → tagging_pending → queued → detecting → tracking → analytics → events → player_highlights → ready`. Emit stage + rough % so the UI's processing screen is honest and calm.
- **Resumability/robustness:** if the server restarts mid-job, status persists; a half-done job can resume or be re-queued. (A 3-hour job dying silently is the worst UX.)

## Pipeline invocation (wrapping the existing scripts)
- Each deliverable maps to existing scripts. The worker runs the shared foundation once (detection → tracking → homography from calibration → teams → ball), then the requested deliverable stages:
  - `coach_analytics` → existing analytics + PDF/video scripts, reading calibration + (sport) config
  - `event_highlights` → existing event-detection + ranked-clipping scripts
  - `player_highlights` → involvement+presence detection → emits clips needing tags → (waits for tags) → assemble per-player reels
- **The tagging dependency:** player_highlights needs human tags MID-pipeline. So that stage runs in two parts: (1) generate involvement/presence clips + expose them via `/tagging-clips`, (2) after the operator submits tags, assemble the named reels. The state machine must support a `tagging_pending` pause.
- Invoke scripts as subprocesses or imported functions; capture their output files into `outputs/`. Surface script failures as a friendly `failed` status + log (never a raw stack trace to the user).

---

## Honest constraints to build for
- **Large files:** match videos are multi-GB. Stream uploads, don't load into memory; store to disk. Show upload progress.
- **Long jobs:** ~3 hours/match on the 4060 (Day-29 number). The whole async/status design exists for this. Set UI expectations honestly via stage progress.
- **Single GPU:** one job processes at a time; queue the rest. Don't run concurrent GPU jobs.
- **Local-network reachability:** server binds to the LAN so other devices reach it (e.g. `0.0.0.0:port`); document how the operator finds the URL (laptop's local IP). Keep it simple + secure-enough for a school LAN (no public exposure).
- **Plain-English errors:** every failure path returns a friendly message; log technical detail server-side only.

## Do NOT
- Do NOT reimplement any CV/ML logic — wrap the existing scripts; the backend is orchestration only.
- Do NOT block web requests on processing — everything long is a background job with status polling.
- Do NOT build cloud/heavyweight infra — local laptop, single GPU, one-job-at-a-time, lightweight queue.
- Do NOT expose technical errors to the operator — friendly messages, server-side logs.
- Do NOT hardcode SoccerNet/dataset paths — the job-config-file is the only input source (finish the Day-31 decoupling).
- Do NOT change field names from the contract — the frontend collects against these exact fields.
